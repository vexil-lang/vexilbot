# Installation Store Persistence Design

**Date:** 2026-03-29
**Status:** Approved

## Overview

The `installationStore` in `cmd/vexilbot/main.go` maps `owner/repo` â†’ GitHub App installation ID. It is populated on each incoming webhook. Currently it is purely in-memory, so after a container restart the dashboard's repo dropdown is empty and all GitHub API calls fail until a new webhook arrives.

This change persists the installation store to a `.vxb` file using the existing vexstore pattern, so the full map (repo list + installation IDs) is recovered on startup.

## Schema

New file `schemas/installation.vexil`:

```vexil
namespace installation

message InstallationEvent {
    0: owner           string
    1: repo            string
    2: installation_id u64
}
```

Each record represents one observation of a repo's installation ID. Multiple records for the same repo are valid (e.g. after a reinstall); replay takes the latest value.

## Storage

`data/installations.vxb` â€” opened on startup with `installation.SchemaHash`, same pattern as `logs.vxb` and `events.vxb`.

## Write Path

`installationStore.set(owner, repo string, id int64)` is called on every incoming webhook. After updating the in-memory map (synchronously, as today), it appends an `InstallationEvent` record to the vxb store. The append is best-effort: errors are logged but do not fail the webhook handler or the in-memory update.

## Replay on Startup

A new `loadFromStore(store *vexstore.AppendStore) error` method on `installationStore` replays all records from the file before the webhook server starts. Records are processed in order; the last record for each `owner/repo` wins (handles reinstalls where the installation ID changes). Errors during individual record decode are skipped (logged); a file read error is returned and logged as a warning in `main()` â€” the server still starts with an empty map.

## Code Changes

### `installationStore` (in `cmd/vexilbot/main.go`)

Add a `store *vexstore.AppendStore` field. All existing methods are unchanged except `set`, which gains the append side-effect. The store field is nil-safe â€” if nil, `set` behaves exactly as today (in-memory only).

```go
type installationStore struct {
    mu      sync.RWMutex
    entries map[string]int64
    store   *vexstore.AppendStore // nil = in-memory only
}

func (s *installationStore) set(owner, repo string, id int64) {
    s.mu.Lock()
    s.entries[owner+"/"+repo] = id
    s.mu.Unlock()
    if s.store != nil {
        ev := &installation.InstallationEvent{
            Owner:          owner,
            Repo:           repo,
            InstallationID: uint64(id),
        }
        bw := vexil.NewBitWriter()
        if ev.Pack(bw) == nil {
            if err := s.store.Append(bw.Finish()); err != nil {
                slog.Error("installation store append", "error", err)
            }
        }
    }
}

func (s *installationStore) loadFromStore(store *vexstore.AppendStore) error {
    records, err := store.ReadAll()
    if err != nil {
        return err
    }
    s.mu.Lock()
    defer s.mu.Unlock()
    for _, rec := range records {
        r := vexil.NewBitReader(rec)
        var ev installation.InstallationEvent
        if err := ev.Unpack(r); err != nil {
            slog.Warn("installation store: skip bad record", "error", err)
            continue
        }
        s.entries[ev.Owner+"/"+ev.Repo] = int64(ev.InstallationID)
    }
    return nil
}
```

### `main()` (in `cmd/vexilbot/main.go`)

Open `installations.vxb` after the other stores, call `loadFromStore`, pass the store to `installationStore`:

```go
installStore, err := vexstore.OpenAppendStore(cfg.Server.DataDir+"/installations.vxb", installation.SchemaHash)
if err != nil {
    fmt.Fprintf(os.Stderr, "open installation store: %v\n", err)
    os.Exit(1)
}
defer installStore.Close()

store := &installationStore{entries: make(map[string]int64), store: installStore}
if err := store.loadFromStore(installStore); err != nil {
    slog.Warn("load installation store", "error", err)
}
```

## Testing

- Unit test for `loadFromStore`: write several records including duplicates for the same repo, verify latest wins.
- Unit test for `set` with a real store: verify records appear in `ReadAll`.
- Nil store: verify `set` and the existing `get`/`list` still work with `store: nil` (regression guard).

## Non-Goals

- No migration of existing deployments â€” the first run after deploy creates a fresh `installations.vxb`. The map repopulates naturally as webhooks arrive. This is acceptable since the store was empty after every restart anyway.
- No compaction â€” the file grows by one record per webhook indefinitely. At typical webhook rates (tens per day) this is negligible.
